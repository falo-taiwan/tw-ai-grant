# 雙層爬蟲中，遍歷詳細頁解析核心內容的 Python 範例
def parse_detail_page(detail_url):
    res = requests.get(detail_url)
    soup = BeautifulSoup(res.content, 'html.parser')
    
    # 提取公司與統編等結構化文字
    company_box = soup.select_one("div.service-content-box-data")
    tax_id = ""
    if company_box:
        comp_text = company_box.text
        # 使用正則表達式抓取台灣 8 碼統編
        match = re.search(r'統一編號：\s*(\d{8})', comp_text)
        if match:
            tax_id = match.group(1)
            
    return {
        "tax_id": tax_id,
        "detail_text": soup.select_one("div.service-content-box").text.strip()
    }
Module 04

資料清洗與正規化 (Data Cleaning & Regex)

網頁抓下來的原始字串極其混亂。例如價格寫著：方案金額 : 102,375 (新台幣/含稅) 軟體使用月數 : 6 (月)。我們必須使用正規表示式 (Regex) 抽離出純數字，以供 RAG 做範圍過濾。

regex_clean.py
import re

# 1. 數值化價格金額
def extract_price_amount(price_str):
    # 移除逗號
    price_clean = price_str.replace(",", "")
    match = re.search(r'\d+', price_clean)
    return int(match.group(0)) if match else 0

# 2. 數值化授權月數
def extract_duration_months(price_str):
    # 匹配「12個月」或「6月」
    match = re.search(r'(\d+)\s*(?:個月|月)', price_str)
    if not match:
        # 匹配「使用月數 : 6」
        match = re.search(r'使用月數\s*:\s*(\d+)', price_str)
    return int(match.group(1)) if match else 12 # 預設12個月

# 驗證
print(extract_price_amount("方案金額 : 102,375 (新台幣)")) # 102375
print(extract_duration_months("方案期程為 12 個月"))      # 12

Gemini API 脈絡快取 (Context Caching) 實戰

在大模型應用中，每次對話都重新上傳 10 萬字的 JSON 會導致高昂的輸入成本與嚴重的解析延遲。Gemini 獨家提供的 Context Caching (脈絡快取)，允許開發者將完整的資料庫一次性載入伺服器記憶體中，後續查詢自動調用，能將 API 成本降低 80% 以上！

快取成本效能計算

以我們的 10 萬 Tokens 資料庫為例，單次詢問的輸入成本將從新台幣 0.25 元降低至 0.07 元，且大模型對全域資料的檢索精準度為 100%，是極具生產價值的商業架構。

# Python 建立與呼叫 Gemini 脈絡快取範例
import datetime
import google.generativeai as genai
from google.generativeai import caching

# 建立快取資源 (TTL 設為 1 小時)
db_cache = caching.CachedContent.create(

Module 07

在地端 SLM 部署的挑戰與抉擇

許多企業基於資安考量希望「完全地端化運行」，我們需要引導學生評估地端輕量化模型（SLM）的硬體極限與架構抉擇：

  • Gemini Nano (1.8B~3.2B)：上下文長度極短（2K~8K），完全無法處理 10 萬 Tokens 的資料庫，推理能力亦不足以進行複雜的跨工具對比。
  • Ollama SLM (Llama 3 8B / Qwen 2.5 7B)：雖然架構支援 128K 上下文，但處理 10 萬 Tokens 的 KV Cache 會吃掉 16GB 以上的顯示記憶體 (VRAM)。在一般筆電上會被迫使用 CPU 運算，導致生成速度降至每秒小於 1 個字，變得完全無法使用。

在地端最佳實踐：混合式過濾法 (Hybrid Filtering)

要在本地端（Local / SLM）流暢運行本專案，必須使用「Python 規則篩選 + Ollama 本地推理」。先用 Python 快速過濾出符合預算與行業的 5 筆工具（總長約 3,000 Tokens），再將這 5 筆資料交給地端的 Llama 3 8B 或 Qwen 2.5 進行語意撰寫，即可在不到 6GB 顯存下實現秒級回覆！

💎 進階架構：前端 RAG 降噪引擎與 4,096 Token 物理超載防禦

在 V2.00 儀表面板中，我們實作了一套業界級的「RAG 降噪與超載防禦工程沙盒」，引導學生在瀏覽器端直觀調優 RAG 性能與成本：

  • 4,096 Tokens 物理壁壘：本地 SLM (如 Gemma 2) 的默認 Context Window 為 4K。若直接發送全量資料（237筆，共 20K+ tokens），會觸發大模型自動截斷，導致輸入佔滿 4,095 tokens，使 AI 僅剩 1 token 可用於輸出而卡死。
  • 5 大前端 RAG 降噪引擎：提供「無過濾」、「欄位投影與精簡（Token 壓縮）」、「TF-IDF 相關性排序」、「語意特徵與同義詞擴展」、「UI 條件聯動過濾」等技術。
  • 純 JS 字元級 TF-IDF 排序與語意同義詞擴展：針對中文採用 Unigram（單字）與 Bigram（雙字）的 N-gram 切分法計算字頻與 IDF 得分，配合行業同義詞自動擴充（如將 POS 與結帳、收銀關聯），在前端極速（<5ms）完成粗篩，將 Token 壓縮至 ~800 tokens，性能提升數十倍。
  • 發送前預檢與攔截 (Pre-flight Safeguard)：發送前以 字元數 * 0.45 高精度估算 Token 數，若超載則強力攔截並探出「超載預警對話框」，提供套用 TF-IDF 自動優化、堅持發送全文（作為超載崩潰對照組）與取消等 HCI 互動方案。
  • 中繼數據足跡 (Metadata Footprint)：為保證報告的可重複性與商業審計，將該次應用的 RAG 引擎名稱寫入本地歷史資料庫，並同步寫入導出的 Markdown 與 Notion 風格 HTML 文件中。

⚠️ 瀏覽器安全性障礙：地端 CORS 與 Mixed Content 防線

在實作「前端網頁直接連線地端/區網 Ollama (如 http://127.0.0.1:11434 或 http://192.168.31.202:11434)」時，學生會遇到瀏覽器的安全攔截而導致 Failed to fetch 錯誤。主要原因與解決方案如下：

  1. 跨來源資源共用 (CORS) 限制：Ollama 預設不允許網頁跨來源連線。必須在啟動 Ollama 前，於伺服器端主機設定環境變數 OLLAMA_ORIGINS="*" 並重啟 Ollama，否則瀏覽器會拒絕載入回應。
  2. 混合內容阻擋 (Mixed Content Block)：如果網頁佈署在 GitHub Pages 等安全網站（網址為 https://），瀏覽器出於安全規定，會嚴格阻擋向地端或區網非安全通訊協定（http://）發送的任何 Fetch 請求。
  3. 繞過與連線解決策略：
    • 本地開啟 (最推薦)：改用 file:// 協議雙擊開啟本地專案目錄中的 index.html 或 v2.html，此時不受混合內容限制。
    • 啟動本地網頁伺服器：在專案目錄下執行 python3 -m http.server 7777，透過 http://localhost:7777 訪問網頁，避開瀏覽器對本地檔案的特殊權限限制。
    • 建立安全加密隧道 (ngrok)：在 Ollama 伺服器主機執行 ngrok http 11434，將服務映射為隨機安全網址（如 https://xxxx.ngrok-free.app），複製該網址填入網頁 IP 欄位，即可在 GitHub Pages 線上版直接安全連線！

⚠️ 深度案例分析：本專案資料源之資安與防爬蟲漏洞剖析

本專案採集的兩個政府補助工具庫網站，在業界資安防禦標準中，防護與防爬蟲（Anti-Crawling）設計均存在顯著漏洞：

  • 來源一（公開 Google Sheet 漏洞）：將底層資料庫直接存放在公開的 Google Sheet 中，並在前端 JS 代碼中暴露了 Sheet ID。這導致政府伺服器對爬蟲流量完全失去感知與監控能力，無法實施 IP 限速或封鎖防禦，且極易外洩內部不慎放入試算表的敏感聯絡資訊。
  • 來源二（自增整數 ID 與掃庫漏洞）：詳細頁使用遞增 ID（如 no=35）且未設置任何速率限制（Rate Limiting）。這使得爬蟲可利用簡單迴圈進行「暴力遍歷掃庫（ID Enumeration）」，並可能引發 BOLA（失效的物件層級授權） 漏洞，越權存取商家未公開的申請草稿與個資。
  • 資安防禦最佳實踐（API 安全四道防線）：
    1. 第一道防線 (WAF)：部署 Cloudflare，識別爬蟲特徵與異常頻率並自動彈出人機驗證（reCAPTCHA）或封鎖。
    2. 第二道防線 (動態簽章)：前端異步請求須攜帶動態計算的加密令牌與時間戳，防止爬蟲直連後台 API。
    3. 第三道防線 (UUID 隨機化)：將自增 ID 改為隨機 UUID，徹底瓦解爬蟲的遍歷計畫。
    4. 第四道防線 (敏感個資遮蔽)：對 Email、電話與公司統編等個資進行遮蔽（如 te**@company.com），通過人機驗證後才延遲加載。

A. 使用 Selenium 模擬真實用戶點擊並等待加載

正確答案為 B。 解析：逆向工程直連 API 不僅速度極快、耗用系統資源最少，還能避開動態渲染帶來的防爬機制與載入噴錯，是專業資料工程師的首選策略。
問題 2：要從字串 "方案價格為 NTD 150,000 / 12個月" 中，使用 Python Regex 匹配出「12」個月，以下哪組正則最合適？
A. r'\d+'
B. r'(\d+)\s*(?:個月|月)'
C. r'/[A-Za-z]+'
正確答案為 B。 解析：如果使用 `\d+`，會首先匹配到 `150000`。因此必須使用 `(\d+)\s*(?:個月|月)` 這樣的特定模式來精準抓取代表月數的數字。
