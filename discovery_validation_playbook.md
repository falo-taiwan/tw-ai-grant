# AI APP Studio：需求發掘與驗證手冊

本手冊提供了一套系統化的方法，引導個人或組織從「日常痛點」中挖掘高價值需求，驗證這些需求是否為「真需求」，並通過「商業分流決策模型」決定這些創意最適合的落地路徑。

---

## Part 1: 痛點挖掘框架 (Problem Discovery)

許多失敗的產品都源於「手握錘子（AI 技術），尋找釘子（問題）」。本框架要求我們回到原點，從**「痛點」**出發。請引導學員或您的團隊，針對日常工作與生活進行以下 **6 大維度** 的系統化掃描。

```
                       ┌────────────────────────┐
                       │   日常與工作痛點掃描器   │
                       └───────────┬────────────┘
         ┌─────────────┬───────────┼───────────┬─────────────┐
   ┌─────┴─────┐ ┌─────┴─────┐ ┌───┴───┐ ┌─────┴─────┐ ┌─────┴─────┐ ┌─────┴─────┐
   │ 1.最花時間 │ │ 2.最常重複 │ │ 3.最煩 │ │ 4.最易遺忘 │ │ 5.最易出錯 │ │ 6.最不想做 │
   └───────────┘ └───────────┘ └───────┘ └───────────┘ └───────────┘ └───────────┘
```

### 1. 每天最花時間的事情 (Time-Consuming Tasks)
*   **定義**：佔用你每天 30% 以上工作時間，且大部分是低認知價值的活動。
*   **思考問題**：如果突然多出 2 個小時，你會想做什麼？現在是哪件事奪走了這 2 個小時？
*   **範例**：
    *   *工作*：整理跨部門的週報、從數十封 Email 中提取客戶需求並分類。
    *   *生活*：每天花半小時比價外送平台與折價券、在網路上搜尋適合的假日親子活動。

### 2. 最重複的事情 (Highly Repetitive Tasks)
*   **定義**：每天或每週都在做，且動作高度一致、幾乎不需要臨場發揮的機械式工作。
*   **思考問題**：哪件事讓你覺得自己像個「複製貼上機器人」？
*   **範例**：
    *   *工作*：將 ERP 中的銷售數據複製到 Excel 中繪製圖表、發送格式固定的開會通知與提醒。
    *   *生活*：每個月固定的水電費、訂閱服務繳費與對帳。

### 3. 最煩的事情 (Annoying Tasks)
*   **定義**：本身不一定花時間，但每次做都會極大消耗你的情緒能量（Mental Energy），讓你產生抗拒心理。
*   **思考問題**：行事曆上哪一個行程或任務只要出現，你就會想「嘆氣」？
*   **範例**：
    *   *工作*：撰寫冗長且充滿官樣文章的績效自評、與不講理的客戶進行第一線的糾紛退換貨溝通。
    *   *生活*：在海量的網路評價中篩選真正誠實的餐廳評論、處理複雜的垃圾郵件退訂。

### 4. 最容易忘記的事情 (Easy-to-Forget Tasks)
*   **定義**：重要但不緊急，或者頻率較低，導致大腦無法形成習慣，極度依賴外部提醒的事情。
*   **思考問題**：上一次因為忘記某件事而導致重大損失或尷尬，是什麼時候？
*   **範例**：
    *   *工作*：定期更換系統密碼、在專案截止日前 3 天提醒外部合作夥伴交付初稿。
    *   *生活*：按時吃藥或補充保健品、在訂閱服務免費試用期結束前取消訂閱。

### 5. 最容易出錯的事情 (Error-Prone Tasks)
*   **定義**：細節極多、邏輯繁瑣、需要反覆人工校對，一旦分神就會產生連鎖性錯誤的工作。
*   **思考問題**：每次交付某件工作前，你得深呼吸並檢查三遍以上，是哪件事？
*   **範例**：
    *   *工作*：手動登錄包含多個品項與稅率的海外採購發票、跨時區的會議時間排定。
    *   *生活*：申報個人綜合所得稅、安排多人旅遊的交通與住宿時間接軌。

### 6. 最不想做的事情 (Undesirable Tasks)
*   **定義**：明知有價值，但因為門檻高、痛苦指數高，導致你不斷拖延（Procrastinate）的事情。
*   **思考問題**：你的待辦清單上，哪件事被移到了下一週超過三次？
*   **範例**：
    *   *工作*：整理堆積了一整季的發票與收據進行報銷、為寫好的代碼撰寫說明文檔。
    *   *生活*：整理塞滿雜物的衣櫃、為家裡的資產與保單做一次全面的整理。

## Part 2: 需求驗證機制 (Demand Validation)

當我們列出了一堆痛點後，必須冷酷地進行篩選。並非所有的痛點都值得被解決。我們需要通過 **「真偽需求診斷法」** 來驗證。

### 1. 真偽需求診斷指標

一個值得被 AI 解決的真需求，必須同時滿足以下四個指標：

```
                    【 真需求 ＝ 痛苦夠深 ＋ 頻率夠高 ＋ 願意付費 ＋ AI 適配 】
```

1.  **痛苦夠深 (Pain Severity)**
    *   *檢驗標準*：如果不解決這個問題，使用者會面臨罰款、損失金錢、加班、或者極大的焦慮嗎？這是一個「止痛藥（Painkiller）」，還是一個「維他命（Vitamin）」？
    *   *黃金問題*：使用者目前為了應對這個問題，付出了多少代價（手動繞道、買其他工具、甚至僱用專人）？
2.  **頻率夠高 (Frequency)**
    *   *檢驗標準*：這個痛點是每天發生、每週發生，還是每年只發生一次？
    *   *評估*：頻率越高，產品的黏著度越高，AI 訓練與優化的數據回饋也越快。每年只發生一次的痛點（如買房子），除非單次價值極高，否則難以支撐一個獨立產品。
3.  **願意付費 (Willingness to Pay)**
    *   *檢驗標準*：目標用戶是否擁有預算，且願意為「省下時間」或「減少痛苦」掏出錢包？
    *   *評估*：企業端（ToB）通常對「節省成本、提高產出、合規防損」有明確的付費意願；個人端（ToC）則對「消除即時焦慮、提升個人競爭力、獲得獨特體驗」更願意付費。
4.  **AI 適配度 (AI Suitability)**
    *   *檢驗標準*：這個問題是否需要「語意理解、生成、推理、模糊匹配或預測」？
    *   *反例*：如果一個問題可以用簡單的 **「Excel 公式」** 或 **「幾行傳統 If-Else 程式碼」** 100% 精確解決，就絕對不要使用 AI。AI 具有隨機性與成本，不適合解決邊界極度清晰的傳統計算問題。

## Part 3: 商業分流決策框架 (Triage Decision Framework)

發想出 AI APP 構想後，我們不應該盲目地直接去寫程式或開源創辦公司。我們必須根據**「數據敏感性、流程變動頻率、商業價值與系統相依性」**，將創意分流到以下三個賽道：

```
                            ┌────────────────────────┐
                            │     AI 創意分流決策      │
                            └───────────┬────────────┘
                                        │
         ┌──────────────────────────────┼──────────────────────────────┐
         ▼                              ▼                              ▼
 ┌───────────────┐              ┌───────────────┐              ┌───────────────┐
 │   ERP 整合軌   │              │   FALO 平台軌  │              │   獨立創業軌  │
 ├───────────────┤              ├───────────────┤              ├───────────────┤
 │ 核心業務數據  │              │ 靈活運營流程  │              │ 跨產業普適性  │
 │ 強一致性要求  │              │ 快速變動需求  │              │ 高額付費意願  │
 │ 嚴格權限控管  │              │ 輕量級自動化  │              │ 獨立 SaaS 潛力│
 └───────────────┘              └───────────────┘              └───────────────┘
```

### 1. ERP 整合軌 (ERP Integration Track)
*   **定義**：將 AI 能力作為功能模組，嵌入到企業既有的 ERP（企業資源規劃）或其他核心事務系統中。
*   **適合特徵**：
    *   高度依賴企業內部核心數據（如：真實財務帳目、主客戶資料、即時庫存）。
    *   對數據的**「一致性」與「精確度」**要求極高，容錯率低。
    *   屬於標準業務流程的自然延伸。
*   **典型案例**：*AI 應收帳款對帳小幫手*。它需要讀取 ERP 的傳票與銀行對帳單，自動進行語意模糊比對，並直接在 ERP 中產生過帳分錄。

### 2. FALO 平台軌 (Flexible & Agile Low-code Operations Track)
*   **定義**：將創意納入企業內部的靈活運營/自動化工作流平台（如 FALO、企業內部 low-code 平台、Dify 企業版）。
*   **適合特徵**：
    *   流程變動頻繁，需要快速迭代以應對業務變化。
    *   主要處理「非結構化資料」（如：客戶詢價 Email、社群合約草稿、產品圖片）。
    *   跨部門的協同運作，但不直接修改 ERP 中的核心財務/庫存硬性數據。
*   **典型案例**：*跨部門合約審查工作流*、*自動化競品價格監控與即時 LINE 報警系統*。

### 3. 獨立創業軌 (Independent Startup Track)
*   **定義**：該創意解決的是一個**跨行業、跨公司、普適性強**的痛點，具備發展成獨立 SaaS 產品並在市場上銷售的潛力。
*   **適合特徵**：
    *   不相依於任何單一企業的特定 ERP 系統後端，只需標準的 API 即可運作。
    *   市場規模（TAM）足夠大，用戶痛點極其強烈，有明確的訂閱制或計量付費空間。
    *   產品本身具備高度的可擴充性（Scalability）。
*   **典型案例**：前面提到的 *夢境繪本師 (DreamWeaver Kids)*、*聲入人心 (AccentShift AI)*，或針對特定專業人士的 *契約之眼 (ContractEye)*。

## Part 4: 實作練習：分流決策自我提問清單

在進行分流時，請對每一個 AI APP 創意詢問以下三個問題：

1.  **「這個 APP 需要實時修改公司的核心財務帳目或庫存數據嗎？」**
    *   如果是 → **歸入 ERP 整合軌**。安全與一致性是第一要務。
2.  **「這個流程是我們公司特有的，且下個月業務規則可能就會變嗎？」**
    *   如果是 → **歸入 FALO 平台軌**。利用低代碼與工作流引擎保持極大靈活性。
3.  **「如果把這個 APP 賣給我們行業內的其他 100 家公司，他們會面臨一模一樣的痛點且可以直接開箱使用嗎？」**
    *   如果是 → **歸入 獨立創業軌**。這代表它具備獨立的商業孵化價值。
