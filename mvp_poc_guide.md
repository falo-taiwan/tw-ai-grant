# AI APP Studio：MVP 與 PoC 規劃指南

本指南旨在引導團隊在不編寫傳統程式碼的情況下，如何快速規劃並建置 AI 應用的**最小可行性產品（MVP, Minimum Viable Product）**，並設計**概念驗證（PoC, Proof of Concept）**流程，以最低的成本、最快的速度在真實環境中驗證技術可行性與用戶價值。

---

## Part 1: 無程式碼/低程式碼 AI MVP 規劃法

在 AI 時代，建置 MVP 的速度比過去快了十倍。藉由**「大語言模型（LLM）＋工作流自動化平台」**，我們可以在幾天甚至幾小時內，將一個點子轉化為可實際體驗的產品。

### 1. 核心開發工具堆疊 (No-code / Low-code Stack)

為了快速搭建 AI MVP，推薦使用以下無需程式設計背景即可上手的工具鏈：

```
   【大語言模型 API】       【AI 應用編排平台】       【工作流自動化】        【前端使用者介面】
┌─────────────────────┐   ┌─────────────────────┐   ┌─────────────────┐   ┌─────────────────┐
│ • OpenAI ChatGPT    │ → │ • Dify (推薦)        │ → │ • Make (原Inte- │ → │ • LINE Bot (最易)│
│ • Claude (Anthropic)│   │ • Coze (字節跳動)   │   │   gromat)       │   │ • Notion Page   │
│ • Gemini (Google)   │   │   (可視化流程與Agent)│   │ • Zapier        │   │ • Softr / Glide │
└─────────────────────┘   └─────────────────────┘   └─────────────────┘   └─────────────────┘
```

*   **AI 應用編排平台 (Dify / Coze)**：
    *   *用途*：這是你的「AI 大腦與流程設計室」。你可以在這裡用拉線（可視化）的方式，設計 AI 該用什麼 Prompt、讀取哪些上傳的 PDF 知識庫（RAG）、在什麼步驟呼叫什麼 API 工具。
*   **工作流自動化平台 (Make / Zapier)**：
    *   *用途*：這是你的「數位管道與手腳」。它可以幫你把不同的 App 串接起來。例如：當「LINE 收到新訊息」→ 傳送給「Dify 處理」→ 將 Dify 的回答「寫入 Notion 並寄送 Email」。
*   **前端展示介面**：
    *   *LINE Bot / Slack Bot*（極力推薦）：最容易上手的介面，使用者只需像聊天一樣輸入資訊，即可體驗 AI 服務。
    *   *Softr / Glide*：能快速將 Google Sheets 或 Notion 轉化為精美網頁/手機 Web App 的無程式碼工具。

### 2. MVP 的核心思維：以「認知」為中心
傳統 MVP 需要開發註冊、登入、付費、後台管理等繁瑣功能。**AI MVP 必須完全剝離這些非核心功能**。
*   **原則**：只驗證 **「AI 能否在特定的輸入下，產生令用戶驚豔的輸出值」**。
*   **作法**：如果你的目標是用戶端網頁，MVP 階段可以先做一個 **LINE 官方帳號**；如果你的目標是自動化報表，MVP 階段可以由你手動接收用戶 Email，丟入你設計的 Dify 工作流，再手動把結果寄回給用戶（這被稱為「綠野仙蹤 MVP」或「Wizard of Oz MVP」，用人工模擬後端系統）。

## Part 2: AI APP 規格定義書範本 (MVP Specification)

在動手建置前，團隊必須為發想的 AI APP 填寫這份規格定義書，確保所有人對產品的核心範疇達成共識。

### 【AI APP 規格定義書】

| 欄位項目 | 詳細說明與填寫指引 |
| :--- | :--- |
| **1. 產品名稱 (Name)** | 為你的 AI APP 取一個直觀、好記、具有功能隱喻的名稱（例如：合約之眼、小商小編）。 |
| **2. 目標使用者 (User)** | 誰是這個產品的第一批核心用戶？（請具體描述，如：個人甜點工作室老闆、跨國專案經理）。 |
| **3. 使用者痛點 (Pain)** | 使用者在沒有這款產品前，面臨最痛苦、最花時間、最容易出錯的具體場景是什麼？ |
| **4. 核心功能 (Core Feature)** | 產品的首發版本只做**「哪一件最核心的事」**？（請勿寫超過 3 個功能）。 |
| **5. 產品輸入 (Input)** | 使用者需要提供什麼資料給 AI？（例如：一張照片加上一句大白話、一個 PDF 報告檔）。 |
| **6. 產品輸出 (Output)** | AI 處理後，呈現給使用者的具體結果是什麼？（例如：三份不同風格的社群文案、一份結構化的中文摘要與任務清單）。 |
| **7. 成功指標 (Metrics)** | 如何衡量這個 MVP 是成功的？（例如：處理時間從 2 小時降至 2 分鐘、產出的文案修改率小於 20%）。 |
| **8. MVP 實作方案 (MVP Scope)** | 我們將用哪些無程式碼工具（如 Dify + LINE Bot）在 3 天內搭建出來？核心流程如何運作？ |
| **9. 未來版本規劃 (Roadmap)** | 如果 MVP 驗證成功，未來的 V1.0、V2.0 版本會加入哪些功能？（如：串接 ERP、多帳號權限管理、付費訂閱系統）。 |

## Part 3: PoC（概念驗證）路徑圖

**PoC（Proof of Concept）** 的目的是回答一個技術問題：**「以目前的 AI 技術與現有數據，到底做不做得到我們想要的精度？」** 

許多 AI 專案失敗是因為 LLM 經常會「胡說八道（Hallucination）」或無法穩定遵循複雜指令。以下是 5 個步驟的 PoC 實作路徑：

```
【步驟 1：準備黃金測試集】 ───► 【步驟 2：設計 Baseline Prompt】 ───► 【步驟 3：極限測試與調優】
                                                                                  │
【步驟 5：商業與技術決策】 ◄─── 【步驟 4：人工盲測與評估 (100分制)】 ◄────────────┘
```

### 步驟 1：準備「黃金測試集」 (Golden Dataset)
*   **執行**：收集 **10 到 20 個** 真實的歷史輸入與對應的「完美人工輸出」案例。
*   **目的**：這組資料將作為你的「標準答案庫」，用來評估 AI 產出的品質。

### 步驟 2：設計 Baseline Prompt (基礎提示詞)
*   **執行**：在 Dify 或 Playground 中，撰寫第一版系統提示詞（System Prompt），明確定義 AI 的角色、任務、限制與輸出格式。
*   **測試**：將測試集輸入 AI，記錄第一版的輸出結果。

### 步驟 3：極限測試與調優 (Prompt Engineering & RAG)
*   **執行**：分析第一版輸出與標準答案的差距。
    *   *如果 AI 缺乏專業背景知識* → 引入 **RAG（檢索增強生成）**，上傳企業手冊或專業領域知識庫。
    *   *如果 AI 格式不受控制* → 使用 **Few-shot Learning**（在 Prompt 中提供 2-3 個輸入輸出的範例格式）。
    *   *如果邏輯容易混亂* → 採用 **CoT（思維鏈提示詞）**，引導 AI 在給出答案前「一步步思考並寫下推理過程」。

### 步驟 4：人工盲測與評估 (Scorecard Evaluation)
*   **執行**：將 AI 的產出與人類專家的產出混在一起，由不參與開發的第三方人員進行評分（採用 [scorecard.md](file:///Users/force/Google_Antigravity/ai_app/scorecard.md) 中的指標）。
*   **標準**：若 AI 的產出在「可用性」上達到人類專家的 **80% 以上**，且產出速度快十倍，則代表技術 PoC 通過。

### 步驟 5：決策與下一步 (Go/No-Go Decision)
*   *通過* → 開始規劃 MVP 的介面包裝（如 LINE Bot 整合），準備進行市場/用戶驗證。
*   *未通過* → 重新評估：是問題難度太高需要改進模型（如換用更強的 Claude 3.5 Sonnet / GPT-4o），還是該問題目前不適合用 AI 解決，需退回重新定義問題。
