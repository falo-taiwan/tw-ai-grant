# AI APP Studio：評審大會評分表與決策矩陣

本文件定義了一套 **100 分制的 AI 產品評估標準（AI APP Scorecard）**。這套標準旨在將天馬行空的「創意」轉化為可量化的「商業決策」，協助評審、企業決策者或創業團隊客觀評估各個 AI APP 構想的落地價值與優先順序。

---

## 1. 評分維度與權重總覽

評分表共包含 **5 大核心維度**，每個維度佔 **20 分**，總分 **100 分**：

```
┌─────────────────────────────────────────────────────────────────────────┐
│                          AI APP 100 分制評估模型                         │
├───────────────┬───────────────┬───────────────┬───────────────┬─────────┤
│ 1. 痛點強度    │ 2. 市場需求    │ 3. AI 適合度   │ 4. 技術可行性  │ 5. 商業 │
│    (20分)     │    (20分)     │    (20分)     │    (20分)     │  價值   │
│               │               │               │               │ (20分)  │
└───────────────┴───────────────┴───────────────┴───────────────┴─────────┘
```

## 2. 評分細則與級距標準 (Grading Rubric)

為了確保不同評審之間的打分一致性，請參考以下級距標準進行評分：

### 1. 痛點強度 (Pain Severity) - 20%
*評估基準：這個問題對目標用戶來說有多痛苦？不解決會怎樣？*
*   **17 - 20 分 (極強烈 - 止痛藥)**：不解決會面臨罰款、重大資金損失、極大焦慮或每日加班 1 小時以上。用戶目前已主動花大錢或用極其繁瑣的手動方式繞道解決。
*   **12 - 16 分 (強烈)**：造成日常工作或生活的不便與效率低下，用戶經常抱怨，但勉強可以忍受。
*   **7 - 11 分 (中等 - 維他命)**：屬於「有了很好，沒有也無妨」的改良型需求。
*   **0 - 6 分 (弱 - 偽需求)**：屬於開發者憑空想像的痛點，真實用戶並不在意，或者發生的頻率極低。

### 2. 市場需求與普適性 (Market Demand) - 20%
*評估基準：有多少人面臨這個痛點？市場規模夠大嗎？*
*   **17 - 20 分 (大眾普適)**：跨行業、跨職能的普適性痛點。例如：幾乎所有上班族都需要整理會議記錄，所有小商家都需要社群小編。
*   **12 - 16 分 (特定行業通用)**：特定垂直領域（如醫療、法律、製造業督導）的通用痛點，該行業內的所有公司皆面臨此問題。
*   **7 - 11 分 (利基市場)**：僅適用於極少數專業人士或特定組織架構下的特定崗位，市場總人數較少。
*   **0 - 6 分 (極窄市場)**：僅適用於極個別特定公司或個人的極特殊情境，難以進行規模化複製。

### 3. AI 適合度 (AI Suitability) - 20%
*評估基準：這個問題為什麼非用 AI 不可？傳統程式或 Excel 公式能解決嗎？*
*   **17 - 20 分 (AI-Native)**：任務高度依賴**語意理解、非結構化資料處理（文本、語音、圖像）、生成創作或複雜語意推理**。傳統 If-Else 規則完全無法寫死。
*   **12 - 16 分 (AI-Enhanced)**：傳統規則可以解決部分問題，但引入 AI 後能讓體驗產生質的飛躍（例如：更聰明的模糊搜索、自然語言介面）。
*   **7 - 11 分 (AI-Optional)**：傳統程式碼或統計模型即可達到 90% 以上的效果，AI 的加入僅是噱頭，且增加了系統不確定性。
*   **0 - 6 分 (不適合 AI)**：屬於高精確度要求的純計算、純流程控制問題（例如：薪資計算、庫存扣減）。用 AI 反而會因「幻覺」帶來高風險與高成本。

### 4. 技術可行性與 MVP 難易度 (Feasibility & Speed) - 20%
*評估基準：我們能在 3 天內做出可運作的 MVP 嗎？是否面臨嚴重的數據壁壘？*
*   **17 - 20 分 (極易落地)**：利用現有的 LLM API 與無程式碼平台（Dify/Make）即可 100% 實現核心功能，無需收集私有數據，無繁瑣系統對接。
*   **12 - 16 分 (中等難度)**：需要整理並上傳私有知識庫（RAG），或需要調用 2-3 個外部公開 API 進行工作流編排。
*   **7 - 11 分 (挑戰性高)**：需要高度客製化的模型微調（Fine-tuning）、需要極高精確度控制，或必須與高度封閉、無 API 的舊系統進行深度整合。
*   **0 - 6 分 (極難實現)**：目前的 LLM 技術底層無法支持（例如：要求 100% 準確預測未來的股市走勢），或存在嚴重的法規與隱私障礙，數據完全無法取得。

### 5. 商業價值與投資報酬率 (Business Value / ROI) - 20%
*評估基準：解決這個問題能帶來多少金錢回報或成本節省？*
*   **17 - 20 分 (高回報/高省時)**：能直接為企業帶來新營收，或為員工節省 80% 以上的時間，具備明確的 ToB 訂閱制收費空間或極高的內部 ROI。
*   **12 - 16 分 (中高價值)**：能顯著提升工作效率或客戶滿意度，間接降低運營成本，付費意願明確。
*   **7 - 11 分 (中等價值)**：能改善員工或使用者的體驗，但難以直接量化其財務回報，付費意願偏弱。
*   **0 - 6 分 (低價值)**：無法帶來效率提升，亦無商業化付費模式，開發成本遠大於解決問題所帶來的收益。

## 4. 決策分流與優先級矩陣 (Decision Matrix)

打分完成後，我們將總得分與「技術可行性」進行交叉分析，將創意分類至以下 **4 大決策象限**：

```
                    高 ▲ ┌───────────────────┬───────────────────┐
                       │     第二象限      │     第一象限      │
                       │   【戰略性突破】   │   【黃金快速贏家】 │
                       │ (高價值/高實現難度) │ (高價值/低實現難度) │
                       ├───────────────────┼───────────────────┤
        商 業 總 得 分  │     第四象限      │     第三象限      │
                       │     【放棄/暫緩】  │   【輕量型點綴】   │
                       │ (低價值/高實現難度) │ (低價值/低實現難度) │
                    低 ▼ └───────────────────┴───────────────────┘
                         ◀───────────────────────────────────▶
                           高 (技術可行性低)   低 (技術可行性高)
                                技術實現難度 (Feasibility)
```

*   **第一象限：黃金快速贏家 (Quick Wins)**
    *   *特徵*：商業總得分高，且技術可行性極高（實現難度低）。
    *   *決策*：**立即執行 PoC/MVP 規劃**。這是最具性價比的切入點，最容易取得早期成功（Quick Wins）。
*   **第二象限：戰略性突破 (Strategic Bets)**
    *   *特徵*：商業價值與痛點極強，但目前技術實現難度高，或需要深度整合核心 ERP 數據。
    *   *決策*：**納入長期產品藍圖（Roadmap）**。進行階段性技術研發，或規劃在未來 V2.0 版本中實施。
*   **第三象限：輕量型點綴 (Nice-to-Haves)**
    *   *特徵*：商業價值中等偏低，但技術實現極其簡單（例如一兩天就能用 low-code 做完）。
    *   *決策*：**做為次要功能或 FALO 平台的靈活練習項目**，不投入主要研發精力。
*   **第四象限：放棄/暫緩 (Avoid/Kill)**
    *   *特徵*：商業價值低，且技術實現難度極高。
    *   *決策*：**立即放棄或進行大幅度的問題重新定義**。避免浪費寶貴的研發與算力資源。
